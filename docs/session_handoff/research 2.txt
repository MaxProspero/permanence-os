What Marc Lou Made: Overview of Data Market Terminal
DMT is essentially a "Bloomberg Terminal for web analytics," designed to visualize website traffic, revenue, and user behavior in a stock-market-inspired format. It's integrated with Marc's broader analytics platform, DataFa.st (a privacy-focused alternative to tools like Google Analytics that connects visitor data to revenue via Stripe integrations). Key features include:
* Ticker-Style Referrer Tracking: Each traffic source (e.g., X.com, Google, direct visits) is treated like a stock ticker symbol, showing real-time or historical data.
* Candlestick Charts: Displays metrics like visitor counts, session volume, cumulative revenue, and conversions as financial-style charts (e.g., open-high-low-close patterns for daily/weekly trends).
* Comparative Analytics: Side-by-side views of current vs. previous periods, highlighting growth or drops (e.g., February peaks with 46K direct visitors generating $5K revenue, as shown in Marc's X post).
* Real-Time Refresh: Built for obsessive checking, with live updates that mimic a trading floor dashboard—perfect for solopreneurs monitoring indie projects.
* API-Driven: Pulls data from DataFa.st's backend, which supports AI-assisted queries (via something called MCP—Model Context Protocol—for natural language data pulls).
Marc launched DMT publicly around February 2026, sharing it via X and LinkedIn, where it resonated in the indie hacker community for its no-BS approach to making analytics addictive and actionable. It's not a standalone product for sale but a free/demo tool tied to DataFa.st (you need an API key from there to use it fully). How to Replicate This Design
Replicating DMT isn't about copying code (it's proprietary), but building something similar using indie-friendly tools. Aim for a web app that pulls your analytics data and renders it in a terminal-like UI. Here's a step-by-step guide, keeping it simple like a Levels-inspired MVP (build in a weekend, ship fast):
1. Set Up the Backend/Data Source:
   * Use a lightweight analytics API like Plausible, Fathom, or build your own with Stripe for revenue + Google Analytics/Umami for traffic.
   * For real-time: Node.js/Express or PHP (Pieter's fave) to fetch data via cron jobs or WebSockets. Store in a simple database like SQLite or MongoDB.
   * Example: Integrate Stripe webhooks for revenue events and referrer tracking from HTTP headers.
2. Frontend Structure (HTML/JS/CSS):
   * Base HTML: A single-page app (SPA) with a <div> for the "terminal" container. Use vanilla JS for dynamism—no React unless needed.
   * CSS for Terminal Look:
      * Body: background: black; color: limegreen; font-family: 'Courier New', monospace;
      * Tickers: Use flexbox or grid for rows of "symbols" (e.g., XCOM +5% $1200). Add CSS animations for scrolling text (@keyframes marquee).
      * Charts: Library like Chart.js or Lightweight Charts (from TradingView) for candlesticks. Style them minimally: no borders, green/red for up/down.
      * Responsiveness: Keep it desktop-first (like old terminals), but add media queries for mobile.
   * JS Logic:
      * Fetch data via API calls (e.g., fetch('/api/analytics')).
      * Render dynamically: Loop through referrers, generate tickers/charts on refresh.
      * Add a refresh button or auto-poll every 10s for that "obsessive check" feel.
      * Emulate terminal "glitch": Subtle CSS filters or JS for flickering text.
3. Tech Stack (Minimalist Indie Style):
   * Core: HTML5, CSS3, Vanilla JS. Host on Vercel/Netlify for free.
   * Charts: Chart.js (free, lightweight) for candlesticks.
   * Data Handling: If scaling, add a backend like Next.js or Laravel, but start with plain files (Pieter often uses one PHP file for everything).
   * Auth/Keys: Simple API key system, like DataFa.st's.
   * Avoid: Heavy libs like Bootstrap—keep it raw.
4. Build Process (Indie Hacker Way):
   * Prototype Fast: Sketch in Figma/CodePen, then code.
   * Test with Real Data: Hook up your own site/Stripe.
   * Ship & Iterate: Post on X/Indie Hackers for feedback, like Marc did. Add features based on replies (e.g., AI queries if ambitious).
   * Tools/Inspo: Study open-source terminals (e.g., GitHub repos for "web terminal emulator"). Pieter's projects are often dissected on Reddit/r/indiehackers—emulate his "core PHP no MVC" if backend-heavy.
   * Cost: $0 to start; scale with cheap hosting.

Sample Code Snippets: Building a Simple Revenue Dashboard
Inspired by Levels' no-BS style, here's how to hack a basic revenue dashboard MVP. Use vanilla JS for frontend, assume a backend API (e.g., PHP/Node pulling from Stripe). Focus on a ticker for MRR and a Chart.js line graph for historical data. Build in a weekend, host on Vercel.
1. Basic HTML Structure (index.html)
html
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <title>Indie Revenue Dashboard</title>
    <style>
        body { background: black; color: limegreen; font-family: monospace; padding: 20px; }
        .ticker { font-size: 24px; animation: marquee 20s linear infinite; white-space: nowrap; overflow: hidden; }
        @keyframes marquee { 0% { transform: translateX(100%); } 100% { transform: translateX(-100%); } }
        #chart { max-width: 800px; margin: 20px auto; }
    </style>
    <script src="https://cdn.jsdelivr.net/npm/chart.js"></script>
</head>
<body>
    <h1>Live Revenue Ticker</h1>
    <div class="ticker" id="ticker">Loading MRR...</div>
    <canvas id="revenueChart"></canvas>
    <button onclick="refreshData()">Refresh</button>
    <script src="script.js"></script>
</body>
</html>
2. JavaScript for Data Fetch and Rendering (script.js)
Assume an API endpoint /api/revenue returns JSON like: { mrr: 5000, history: [{date: '2026-01', revenue: 4500}, {date: '2026-02', revenue: 5000}] }.
javascript
async function fetchData() {
    const response = await fetch('/api/revenue');
    const data = await response.json();
    return data;
}


async function updateDashboard() {
    const data = await fetchData();
    document.getElementById('ticker').innerText = `MRR: $${data.mrr} ↑5% | Users: 1.2K | Churn: 2% | (Refreshed: ${new Date().toLocaleTimeString()})`;


    const ctx = document.getElementById('revenueChart').getContext('2d');
    new Chart(ctx, {
        type: 'line',
        data: {
            labels: data.history.map(item => item.date),
            datasets: [{
                label: 'Monthly Revenue',
                data: data.history.map(item => item.revenue),
                borderColor: 'limegreen',
                tension: 0.1
            }]
        },
        options: {
            scales: { y: { beginAtZero: true, ticks: { color: 'limegreen' } } },
            plugins: { legend: { labels: { color: 'limegreen' } } }
        }
    });
}


function refreshData() {
    location.reload(); // Or fetch without reload for live feel
}


updateDashboard(); // Initial load
setInterval(updateDashboard, 60000); // Auto-refresh every minute

php
<?php
require 'vendor/autoload.php'; // Composer for Stripe
\Stripe\Stripe::setApiKey('your_stripe_key');


header('Content-Type: application/json');


// Fetch MRR (simplified; in reality, calculate from subscriptions)
$subs = \Stripe\Subscription::all(['status' => 'active']);
$mrr = 0;
foreach ($subs->data as $sub) {
    $mrr += $sub->plan->amount / 100; // Monthly assumption
}


// Mock history (pull from DB in prod)
$history = [
    ['date' => '2026-01', 'revenue' => 4500],
    ['date' => '2026-02', 'revenue' => $mrr]
];


echo json_encode(['mrr' => $mrr, 'history' => $history]);
?>

tmux-resurrect for Persistence
tmux-resurrect is an open-source plugin for tmux (a terminal multiplexer) that enables saving and restoring tmux sessions across system restarts or shutdowns. It captures the full state of your tmux environment, including all windows, panes, layouts, current working directories, and even the contents of panes (like command histories or running processes, if supported by the program). This makes it ideal for maintaining persistent workflows, especially in development scenarios where you might have multiple terminals open for different tasks—such as running servers, editors, or monitoring tools.
Key features include:
* Automatic Restoration Options: When combined with the tmux-continuum plugin, it can auto-save sessions at regular intervals (e.g., every 15 minutes) and restore them on tmux startup, ensuring seamless continuity without manual intervention.
* Process Restoration: It can restart certain programs like vim, neovim, or basic shell commands upon restore, though this is configurable via a whitelist to avoid issues with complex or stateful apps.
* Key Bindings: Default shortcuts are prefix + Ctrl-s to save the current environment and prefix + Ctrl-r to restore the last saved state. No extensive configuration is needed out of the box.
* Storage: Sessions are saved as serialized files in ~/.tmux/resurrect/ (or similar paths), allowing manual rollback to older saves by symlinking to previous files if the latest restore fails.
In the context of multi-agent AI tools like Collaborator AI (from the X post), tmux-resurrect addresses persistence by preserving the state of multiple terminal sessions where AI agents operate. For instance, if agents are collaborating in separate panes (e.g., one building code, another testing), a restart won't lose the setup. However, it's not perfect for all apps—interactive processes might not resume flawlessly, and alternatives like tmuxinator or session managers in tools like Warp could be considered for more advanced needs.

Claude's Skill Guide Explained
Anthropic's "The Complete Guide to Building Skills for Claude" is a 33-page document released in early 2026 that outlines how to create reusable "Skills" for their Claude AI models. What Skills Are and Why They Matter
A Skill is a portable folder you upload to Claude (via Claude.ai, Claude Code, or the API) that teaches it how to execute workflows, enforce rules, or apply expertise. For example, a Skill could automate code reviews, generate reports, or handle compliance checks. Benefits include:
* Consistency and Efficiency: Claude follows the same process every time, reducing errors and saving users from re-explaining details.
* Scalability: Skills work across platforms and can be shared or deployed organization-wide.
* Enhanced Capabilities: They build on MCP by adding "recipes" for using tools intelligently, rather than just providing raw access.
* Composability: Multiple Skills can run together without interference, making Claude adaptable for complex setups.
Skills are an open standard, meaning they're not locked to Anthropic and could potentially work with other AIs in the future.
Anatomy of a Skill
The core is a folder named in kebab-case (e.g., data-analysis-skill). Key elements:
* SKILL.md (Required): The main file with YAML frontmatter at the top (for metadata like name and description) and Markdown body for instructions. The frontmatter includes triggers like "Use when user asks to analyze sales data."
* scripts/ (Optional): Holds executable code (e.g., Python scripts) that Claude can run for automation.
* references/ (Optional): Stores supporting docs or examples loaded only when needed.
* assets/ (Optional): Templates, images, or files for outputs.
No README.md inside the folder—keep that at the repo level for sharing.
Three Levels of Progressive Disclosure
To optimize context and performance, Skills load information in tiers:
1. Level 1 (YAML Frontmatter): Always in Claude's system prompt; minimal details for detecting when to activate (e.g., keywords or phrases).
2. Level 2 (SKILL.md Body): Loaded only if relevant; full step-by-step instructions, examples, and error handling.
3. Level 3 (Linked Files): On-demand access to deeper references or assets, avoiding token overload.
This system ensures Claude stays efficient while accessing details as required.
Writing Effective Instructions
Focus on clarity and structure in SKILL.md:
* Use headers, bullets, and numbered steps for workflows.
* Include examples of inputs/outputs and troubleshooting sections.
* Link to scripts or references explicitly (e.g., "Run python scripts/process.py on the file").
* Emphasize critical rules with bold or special headers.
* Avoid vagueness—be actionable, like specifying exact commands instead of general advice.
Common pitfalls: Overly broad triggers leading to false activations, or instructions buried too deep.
Five Design Patterns
The guide details five architectural patterns for different scenarios, with examples:
1. Sequential Workflow Orchestration: Linear steps, like onboarding a user (create account → setup payment → send confirmation).
2. Multi-MCP Coordination: Integrating multiple services, e.g., pulling from Figma, saving to Google Drive, notifying via Slack.
3. Iterative Refinement: Loops for improvement, such as drafting a report, validating it, then refining based on feedback.
4. Context-Aware Tool Selection: Dynamic choices, like selecting storage based on file type (GitHub for code, Drive for docs).
5. Domain-Specific Intelligence: Embedding expertise, e.g., checking financial regulations before processing a transaction.
These patterns use techniques like data passing between steps, validation checks, and decision trees.

Arbitrage in Crypto Markets
Crypto arbitrage involves exploiting price differences for the same asset across exchanges or markets to lock in risk-free profits. In 2026, with fragmented liquidity and advanced automation, it's a popular strategy, but opportunities are fleeting (often milliseconds) due to market efficiency. Bots are essential for speed, as manual trading can't compete. Profits typically range from 0.1-2% per trade after fees, with sophisticated setups yielding 8-15% monthly returns. Here's an overview, strategies, risks, and tools:
Key Strategies
* Cross-Exchange Arbitrage: Buy low on one exchange (e.g., BTC at $95,000 on Binance) and sell high on another (e.g., $95,100 on Coinbase). Requires accounts and funds on multiple platforms.
* Triangular Arbitrage: On a single exchange, trade three pairs (e.g., BTC/USDT → ETH/BTC → ETH/USDT) to exploit inefficiencies. Faster but needs precise calculations.
* Futures-Spot Arbitrage: Short a higher-priced futures contract while buying spot to capture basis or funding rates. Market-neutral, focusing on differentials rather than direction.
* Funding Rate Arbitrage: In perpetual futures, take opposing long/short positions to collect funding payments (e.g., longs pay shorts when positive). Ideal for low-volatility returns.
* Statistical Arbitrage: Uses historical correlations for mean-reversion trades, often AI-driven for pattern recognition.
In 2026, AI agents monitor spreads and automate across 5-15 exchanges, incorporating risk controls like minimum spread thresholds.

        address initiator,
        bytes calldata params

        address[] memory path = new address[](2);
        path[0] = asset;
        path[1] = token1;

        path[1] = asset;

Ethereum MEV-Boost
MEV-Boost is an implementation of Proposer-Builder Separation (PBS). It allows Ethereum validators to outsource block construction to specialized "Builders" to maximize profit.
   * Proposer (Validator): Sells their block-building rights for a slot.
   * Builder: Aggregates transactions into a full block template designed for maximum MEV.
   * Relay: Acts as a trusted broker. It verifies the blocks from builders and passes only the most profitable "header" to the validator to prevent the validator from stealing the MEV strategy.

## The RBI System — His Master Framework


The foundational strategy he teaches is **RBI: Research → Backtest → Incubate**.


**1. Research**
- Study existing trading strategies from academic papers, quant books, and alpha-generating sources
- Look for strategies that have worked historically and can be expressed in code
- Specific sources: academic quant research, TradingView strategy scripts, crypto-specific indicators


**2. Backtest**
- Use OHLCV (open/high/low/close/volume) historical data to test strategies on the past
- His tool of choice is Python + `backtesting.py` or custom backtesting engines using pandas
- Goal: prove statistical edge before risking real money — "use OHLCV data to backtest the strategy and see if it actually worked in the past"


**3. Incubate**
- If backtested strategy is profitable, run it live with tiny size (paper trade or micro amounts)
- Scale up gradually once it proves itself in real market conditions
- Never skip this phase — it catches the gap between backtested and real performance

## PART 2: THE HYPERLIQUID UTILITY LIBRARY (`nice_funcs.py`)


This is the foundation Moon Dev uses in every bot. It handles all communication with Hyperliquid. Save this as `nice_funcs.py`:


```python
from eth_account.signers.local import LocalAccount
import eth_account
import json
import time
from hyperliquid.info import Info
from hyperliquid.exchange import Exchange
from hyperliquid.utils import constants
import ccxt
import pandas as pd
from datetime import datetime, timedelta
import pandas_ta as ta
import requests
from termcolor import colored


def ask_bid(symbol):
    """Get current ask and bid prices from Hyperliquid orderbook"""
    url = 'https://api.hyperliquid.xyz/info'
    headers = {'Content-Type': 'application/json'}
    data = {'type': 'l2Book', 'coin': symbol}
    response = requests.post(url, headers=headers, data=json.dumps(data))
    l2_data = response.json()['levels']
    bid = float(l2_data[0][0]['px'])
    ask = float(l2_data[1][0]['px'])
    return ask, bid, l2_data


def get_sz_px_decimals(symbol):
    """Get size and price decimal precision for a symbol"""
    url = 'https://api.hyperliquid.xyz/info'
    headers = {'Content-Type': 'application/json'}
    data = {'type': 'meta'}
    response = requests.post(url, headers=headers, data=json.dumps(data))
    symbols = response.json()['universe']
    symbol_info = next((s for s in symbols if s['name'] == symbol), None)
    sz_decimals = symbol_info['szDecimals'] if symbol_info else 0
    ask = ask_bid(symbol)[0]
    ask_str = str(ask)
    px_decimals = len(ask_str.split('.')[1]) if '.' in ask_str else 0
    return sz_decimals, px_decimals


def adjust_leverage_usd_size(symbol, usd_size, leverage, account):
    """Calculate position size in coins from a USD target + leverage"""
    exchange = Exchange(account, constants.MAINNET_API_URL)
    info = Info(constants.MAINNET_API_URL, skip_ws=True)
    exchange.update_leverage(leverage, symbol, is_cross=False)
    price = ask_bid(symbol)[0]
    size = (usd_size / price) * leverage
    rounding = get_sz_px_decimals(symbol)[0]
    size = round(float(size), rounding)
    return leverage, size


def get_ohlcv(symbol, interval, lookback_days):
    """Fetch OHLCV candlestick data directly from Hyperliquid"""
    end_time = datetime.now()
    start_time = end_time - timedelta(days=lookback_days)
    url = 'https://api.hyperliquid.xyz/info'
    headers = {'Content-Type': 'application/json'}
    data = {
        "type": "candleSnapshot",
        "req": {
            "coin": symbol,
            "interval": interval,
            "startTime": int(start_time.timestamp() * 1000),
            "endTime": int(end_time.timestamp() * 1000)
        }
    }
    response = requests.post(url, headers=headers, json=data)
    if response.status_code == 200:
        return response.json()
    return None


def get_position(symbol, account):
    """Get current open position details for a symbol"""
    info = Info(constants.MAINNET_API_URL, skip_ws=True)
    user_state = info.user_state(account.address)
    in_pos = False
    size = 0
    pos_sym = None
    entry_px = 0
    pnl_perc = 0
    positions = []
    for position in user_state["assetPositions"]:
        if (position["position"]["coin"] == symbol and 
            float(position["position"]["szi"]) != 0):
            positions.append(position["position"])
            in_pos = True
            size = float(position["position"]["szi"])
            pos_sym = position["position"]["coin"]
            entry_px = float(position["position"]["entryPx"])
            pnl_perc = float(position["position"]["returnOnEquity"]) * 100
            break
    long = True if size > 0 else (False if size < 0 else None)
    return positions, in_pos, size, pos_sym, entry_px, pnl_perc, long


def limit_order(coin, is_buy, sz, limit_px, reduce_only, account):
    """Place a limit order on Hyperliquid"""
    exchange = Exchange(account, constants.MAINNET_API_URL)
    rounding = get_sz_px_decimals(coin)[0]
    sz = round(sz, rounding)
    order_result = exchange.order(
        coin, is_buy, sz, limit_px,
        {"limit": {"tif": "Gtc"}},
        reduce_only=reduce_only
    )
    return order_result


def cancel_all_orders(account):
    """Cancel all open orders"""
    exchange = Exchange(account, constants.MAINNET_API_URL)
    info = Info(constants.MAINNET_API_URL, skip_ws=True)
    open_orders = info.open_orders(account.address)
    for order in open_orders:
        exchange.cancel(order['coin'], order['oid'])


def pnl_close(symbol, take_profit_pct, stop_loss_pct, account):
    """Close position if TP or SL thresholds are hit"""
    positions, in_pos, size, pos_sym, entry_px, pnl_perc, is_long = get_position(symbol, account)
    if not in_pos:
        return
    if pnl_perc >= take_profit_pct or pnl_perc <= stop_loss_pct:
        ask, bid, _ = ask_bid(symbol)
        close_px = bid if is_long else ask
        limit_order(symbol, not is_long, abs(size), close_px, True, account)
        print(f"Closed {symbol} at {pnl_perc:.2f}% PnL")
```

**The Backtest Script (`backtest/bb_squeeze_adx.py`):**


```python
import pandas as pd
import numpy as np
import talib
from backtesting import Backtest, Strategy


# Load data
data = pd.read_csv('data/BTC-6h-data.csv', parse_dates=['datetime'], index_col='datetime')
data.columns = ['Open', 'High', 'Low', 'Close', 'Volume']


class BBSqueezeADX(Strategy):
    # Strategy parameters (these get optimized)
    bb_window = 20
    bb_std = 2.0
    keltner_window = 20
    keltner_atr_mult = 1.5
    adx_period = 14
    adx_threshold = 25
    take_profit = 0.05   # 5%
    stop_loss = 0.03     # 3%


    def init(self):
        # Bollinger Bands
        self.upper_bb, self.middle_bb, self.lower_bb = self.I(
            talib.BBANDS, self.data.Close, self.bb_window, self.bb_std, self.bb_std
        )
        # ATR for Keltner Channels
        self.atr = self.I(talib.ATR, self.data.High, self.data.Low, self.data.Close, self.keltner_window)
        # Keltner Channels
        self.keltner_middle = self.I(talib.SMA, self.data.Close, self.keltner_window)
        self.upper_kc = self.I(lambda: self.keltner_middle + self.keltner_atr_mult * self.atr)
        self.lower_kc = self.I(lambda: self.keltner_middle - self.keltner_atr_mult * self.atr)
        # Squeeze = BB is inside Keltner (volatility compression)
        self.squeeze = self.I(lambda: (self.upper_bb < self.upper_kc) & (self.lower_bb > self.lower_kc))
        # ADX trend strength
        self.adx = self.I(talib.ADX, self.data.High, self.data.Low, self.data.Close, self.adx_period)
        self.squeeze_released = False


    def next(self):
        squeeze_now = self.squeeze[-1]
        squeeze_prev = self.squeeze[-2] if len(self.data) > 1 else True


        # Detect squeeze ending
        if squeeze_prev and not squeeze_now:
            self.squeeze_released = True


        # Enter on squeeze release with ADX confirmation
        if self.squeeze_released and self.adx[-1] > self.adx_threshold:
            if self.data.Close[-1] > self.upper_bb[-1] and not self.position:
                # LONG breakout
                self.buy(
                    sl=self.data.Close[-1] * (1 - self.stop_loss),
                    tp=self.data.Close[-1] * (1 + self.take_profit)
                )
                self.squeeze_released = False
            elif self.data.Close[-1] < self.lower_bb[-1] and not self.position:
                # SHORT breakdown
                self.sell(
                    sl=self.data.Close[-1] * (1 + self.stop_loss),
                    tp=self.data.Close[-1] * (1 - self.take_profit)
                )
                self.squeeze_released = False


# Run backtest
bt = Backtest(data, BBSqueezeADX, cash=100_000, commission=0.002)
```

```python
"""
🌙 Moon Dev BB Squeeze ADX Live Trading Bot
Trades on Hyperliquid using the strategy proven in backtest
"""


import os, sys, time, schedule, traceback
import pandas as pd
import numpy as np
import talib
import eth_account
from eth_account.signers.local import LocalAccount
from dotenv import load_dotenv
from datetime import datetime
from termcolor import colored


# Add parent dir to path and import utility functions
sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))
import nice_funcs as n


load_dotenv()


# ===== CONFIGURATION — Tune these from your backtest results =====
SYMBOL = 'BTC'
LEVERAGE = 5
POSITION_SIZE_USD = 50       # Start small! $50 to match backtest risk %
BB_WINDOW = 20
BB_STD = 2.0
KELTNER_WINDOW = 20
KELTNER_ATR_MULT = 1.5
ADX_PERIOD = 14
ADX_THRESHOLD = 25
TAKE_PROFIT_PERCENT = 5.0    # From backtest optimization
STOP_LOSS_PERCENT = -3.0
CANDLE_INTERVAL = '6h'
CANDLE_LOOKBACK = 100        # Days of historical candles to fetch


# Initialize Hyperliquid account from private key
HYPER_LIQUID_KEY = os.getenv('HYPER_LIQUID_KEY')
account: LocalAccount = eth_account.Account.from_key(HYPER_LIQUID_KEY)


# Track squeeze state
squeeze_flag = False
squeeze_released = False


def fetch_klines():
    """Fetch OHLCV candles and return as DataFrame"""
    raw = n.get_ohlcv(SYMBOL, CANDLE_INTERVAL, CANDLE_LOOKBACK)
    if not raw:
        return None
    df = pd.DataFrame(raw, columns=['timestamp', 'open', 'high', 'low', 'close', 'volume'])
    df['timestamp'] = pd.to_datetime(df['timestamp'], unit='ms')
    df.set_index('timestamp', inplace=True)
    for col in ['open', 'high', 'low', 'close', 'volume']:
        df[col] = pd.to_numeric(df[col])
    return df


def calculate_indicators(df):
    """Add BB Squeeze + ADX indicators to DataFrame"""
    # Bollinger Bands
    df['upper_bb'], df['middle_bb'], df['lower_bb'] = talib.BBANDS(
        df['close'], timeperiod=BB_WINDOW, nbdevup=BB_STD, nbdevdn=BB_STD
    )
    # ATR
    df['atr'] = talib.ATR(df['high'], df['low'], df['close'], timeperiod=KELTNER_WINDOW)
    # Keltner Channels
    df['keltner_mid'] = talib.SMA(df['close'], timeperiod=KELTNER_WINDOW)
    df['upper_kc'] = df['keltner_mid'] + KELTNER_ATR_MULT * df['atr']
    df['lower_kc'] = df['keltner_mid'] - KELTNER_ATR_MULT * df['atr']
    # Squeeze: BB is inside Keltner = low volatility compression
    df['squeeze'] = (df['upper_bb'] < df['upper_kc']) & (df['lower_bb'] > df['lower_kc'])
    # ADX trend strength
    df['adx'] = talib.ADX(df['high'], df['low'], df['close'], timeperiod=ADX_PERIOD)
    return df


def check_entry_signals(df):
    """Return (long_signal, short_signal) based on squeeze + ADX"""
    current = df.iloc[-1]
    previous = df.iloc[-2]
    squeeze_just_released = previous['squeeze'] and not current['squeeze']
    
    if squeeze_just_released and current['adx'] > ADX_THRESHOLD:
        if current['close'] > current['upper_bb']:
            print(colored('🚀 LONG SIGNAL: BB squeeze released + upward breakout + strong ADX', 'green'))
            return True, False
        elif current['close'] < current['lower_bb']:
            print(colored('📉 SHORT SIGNAL: BB squeeze released + downward breakdown + strong ADX', 'red'))
            return False, True
    return False, False


def bot():
    """Main bot cycle — runs every minute, acts on 6h candles"""
    print(f"\n{'='*60}")
    print(f"🌙 Moon Dev Bot — {datetime.now().strftime('%Y-%m-%d %H:%M:%S')}")
    
    try:
        # Check existing position first
        positions, in_pos, size, pos_sym, entry_px, pnl_perc, is_long = n.get_position(SYMBOL, account)
        
        if in_pos:
            print(f"📊 In position: {SYMBOL} {'LONG' if is_long else 'SHORT'} @ ${entry_px:.2f} | PnL: {pnl_perc:.2f}%")
            n.pnl_close(SYMBOL, TAKE_PROFIT_PERCENT, STOP_LOSS_PERCENT, account)
            return  # Don't look for new entries while in a trade
        
        print("🔍 No position — scanning for entry...")
        n.cancel_all_orders(account)
        
        df = fetch_klines()
        if df is None:
            print("❌ Failed to fetch candle data")
            return
        
        df = calculate_indicators(df)
        long_signal, short_signal = check_entry_signals(df)
        
        if long_signal or short_signal:
            ask, bid, _ = n.ask_bid(SYMBOL)
            lev, pos_size = n.adjust_leverage_usd_size(SYMBOL, POSITION_SIZE_USD, LEVERAGE, account)
            
            if long_signal:
                print(f"🎯 Placing LONG limit order at ${bid:.2f}")
                n.limit_order(SYMBOL, True, pos_size, bid, False, account)
            elif short_signal:
                print(f"🎯 Placing SHORT limit order at ${ask:.2f}")
                n.limit_order(SYMBOL, False, pos_size, ask, False, account)
        else:
            print("⏳ No signals — waiting for squeeze...")
            
    except Exception as e:
        print(colored(f"❌ Bot error: {e}", 'red'))
        print(traceback.format_exc())


def main():
    print("🌙 Moon Dev BB Squeeze ADX Bot Starting...")
    bot()  # Run immediately on start
    schedule.every(1).minutes.do(bot)
    while True:
        try:
            schedule.run_pending()
            time.sleep(1)
        except KeyboardInterrupt:
            print("⚠️ Bot stopped by user")
            break


if __name__ == "__main__":
    main()
```

## PART 8: THE GUARDIAN AGENT (Process Manager)


From the "guardian agent to kill python processes and save RAM" short — this is a meta-agent that watches your other bots.


```python
"""
Guardian Agent — kills runaway processes and monitors RAM
Run this alongside your trading bots
"""
import psutil
import subprocess
import time
import os


# Bots to monitor (process name patterns)
MONITORED_BOTS = ['bb_squeeze_bot.py', 'liquidation_monitor.py']
MAX_RAM_PER_PROCESS_MB = 500  # Kill if a bot uses more than 500MB
RESTART_ON_KILL = True


def get_python_processes():
    """Get all running Python processes"""
    procs = []
    for proc in psutil.process_iter(['pid', 'name', 'cmdline', 'memory_info']):
        try:
            if 'python' in proc.info['name'].lower():
                procs.append(proc)
        except (psutil.NoSuchProcess, psutil.AccessDenied):
            pass
    return procs


def check_and_cleanup():
    """Find and kill runaway processes"""
    for proc in get_python_processes():
        try:
            cmdline = ' '.join(proc.info['cmdline'] or [])
            ram_mb = proc.info['memory_info'].rss / 1024 / 1024
            
            for bot_name in MONITORED_BOTS:
                if bot_name in cmdline:
                    if ram_mb > MAX_RAM_PER_PROCESS_MB:
                        print(f"⚠️ {bot_name} using {ram_mb:.0f}MB — killing and restarting")
                        proc.kill()
                        if RESTART_ON_KILL:
                            subprocess.Popen(['python', bot_name])
        except (psutil.NoSuchProcess, psutil.AccessDenied):
            pass


if __name__ == "__main__":
    print("🛡️ Guardian Agent running...")
    while True:
        check_and_cleanup()
        time.sleep(60)
```

**In one sentence:** It's a Claude Code plugin that takes *any software* (GIMP, Blender, OBS, LibreOffice, your own apps) and automatically generates a complete command-line interface for it so AI agents can control it programmatically — in one command.


The core problem it solves: AI agents are good at reasoning but can't use real professional software because most tools only have GUIs. CLI-Anything bridges that gap by auto-generating a full CLI wrapper with structured JSON output, a REPL mode, tests, and documentation.


---


## How It Works — 7-Phase Auto Pipeline


When you run `/cli-anything <software>` it automatically does all 7 of these:


1. **Analyze** — Scans the source code, maps GUI actions to underlying APIs
2. **Design** — Architects command groups, state model, output formats
3. **Implement** — Builds a Click-based CLI with REPL, JSON output, undo/redo
4. **Plan Tests** — Creates a TEST.md with unit + E2E test plans
5. **Write Tests** — Implements the test suite
6. **Document** — Updates TEST.md with results
7. **Publish** — Creates setup.py, installs to PATH


After running it you get something like `cli-anything-gimp` on your PATH that any AI agent can call.

## How It Actually Works — Step by Step


Every 15 minutes this loop runs:


**Step 1 — Data Collection** (`ohlcv_collector.py`)
Fetches OHLCV candles for every token in `MONITORED_TOKENS` via **Birdeye API**, caches them to temp files, calculates MA20, MA40, RSI indicators on each token.


**Step 2 — AI Analysis** (`trading_agent.py → analyze_market_data`)
Sends each token's market data to **Claude** with this prompt:
```
Analyze: Price vs MA20/MA40, RSI levels, Volume, Recent moves
Respond: BUY / SELL / NOTHING + reasoning + confidence %
```
Results go into a pandas DataFrame with token, action, confidence, reasoning.


**Step 3 — Portfolio Allocation** (`trading_agent.py → allocate_portfolio`)
Sends all BUY recommendations back to Claude with:
```
Given total portfolio size + recommendations,
allocate capital — max 30% per position, keep 20% in USDC
Return as a Python dict: {token_address: dollar_amount}
```


**Step 4 — Exit Positions** (handles SELL/NOTHING first)
Loops through recommendations — any token with a SELL or NOTHING signal where you hold a position gets killed using `chunk_kill()` which sells in 3 equal chunks to avoid slippage.


**Step 5 — Enter Positions** (`ai_entry` → `market_buy`)
Uses **Jupiter DEX API** (`https://quote-api.jup.ag/v6/`) to execute Solana token swaps. Chunks orders into `max_usd_order_size` pieces, retries on failure.

def setup_moon_dev_ecosystem():
    # 1. Define the base directory

    # 2. List of required sibling repositories

        if not repo_path.exists():

            repo_path.mkdir(exist_ok=True)
            # Note: You still need to 'git clone' into these folders

import os

def setup_moon_dev():

    print(f"📝 Created .env with GITHUB_DIR set to: {base_path}")
    print("\n" + "="*40)
    print("✨ NEXT STEPS:")
    print("1. Open your terminal in this folder.")
    for folder, url in repos.items():

To finalize your institutional-grade setup, we need to calculate your Position Size using the Kelly Criterion. This formula helps you determine exactly how much of your bankroll to risk on each trade to maximize growth while avoiding the "Gambler's Ruin" (going to zero).
📐 The Position Size Formula
For prediction markets like Polymarket, we use the simplified version of the formula:
$$K\% = W - \frac{(1 - W)}{R}$$
                     * $K\%$: The percentage of your total bankroll to risk on one trade.
                     * $W$: Your Win Probability (the "Confidence" from your AI Agent or the Leaderboard).
                     * $R$: The Win/Loss Ratio (Profit per $1 risked).
________________

You can add this script to your moon-dev-ai-agents-for-trading folder to automatically update your trade.toml position sizes.

def calculate_kelly_size(bankroll, win_rate, profit_ratio, fraction=0.5):
    """
    bankroll: Total balance (e.g., $1000)
    win_rate: Decimal (e.g., 0.60 for 60%)
    profit_ratio: Potential gain / Potential loss (e.g., 0.90 for Polymarket)
    fraction: 'Fractional Kelly' (0.5 is safer/standard for trading)
    """
    # Kelly Formula
    kelly_percent = win_rate - ((1 - win_rate) / profit_ratio)
    
    # Apply safety fraction and ensure it's not negative
    safe_size_percent = max(0, kelly_percent * fraction)
    dollar_amount = bankroll * safe_size_percent
    
    return round(dollar_amount, 2)


# Example: $1000 bankroll, 65% whale win rate, betting on a 50/50 outcome
print(f"💰 Recommended Trade Size: ${calculate_kelly_size(1000, 0.65, 0.95)}")

import yt_dlp
import whisper
import os


def transcribe_x_video(url):
    # 1. Download only the audio from the X link
    ydl_opts = {
        'format': 'bestaudio/best',
        'outtmpl': 'x_audio.%(ext)s',
        'postprocessors': [{
            'key': 'FFmpegExtractAudio',
            'preferredcodec': 'mp3',
            'preferredquality': '192',
        }],
    }
    
    with yt_dlp.YoutubeDL(ydl_opts) as ydl:
        ydl.download([url])
    
    # 2. Load Whisper (same logic as the Moon Dev strike repo)
    model = whisper.load_model("base")
    print("🎬 Transcribing X content...")
    result = model.transcribe("x_audio.mp3")
    
    # 3. Save as the standard 'video_transcript.txt'
    with open("video_transcript.txt", "w", encoding="utf-8") as f:
        f.write(result["text"])
    
    print("✅ Transcription complete. Check video_transcript.txt")

def analyze_insider_potential(whale_trade_time, intel_publish_time):

def calculate_scaled_position(base_amount, lead_time_minutes, whale_loyalty_score):
    """
    base_amount: Your default trade size from trade.toml
    lead_time_minutes: Minutes whale traded before news (from Comparison Engine)
    whale_loyalty_score: Historical accuracy of the whale
    """
    multiplier = 1.0


    # 1. Scale based on Lead Time (Timing Alpha)
    if lead_time_minutes > 15:
        multiplier += 1.0  # Insider Alert
    elif lead_time_minutes > 5:
        multiplier += 0.5  # High Conviction


    # 2. Scale based on Whale Reliability (Loyalty Score)
    if whale_loyalty_score > 80:
        multiplier += 0.5  # Trusted Whale
    elif whale_loyalty_score < 20:
        multiplier -= 0.5  # High Risk/Laggard


    scaled_amount = base_amount * multiplier
    return round(scaled_amount, 2)

def get_dynamic_weight(defense_score, momentum_score, preference_slider=0.5):
    """
    preference_slider: 0.0 (Pure Momentum) to 1.0 (Pure Value/Defense)
    Adapted from roverchen/sinopac-quant-app logic.
    """
    # Combine scores based on your 'Risk Preference'
    final_score = (preference_slider * defense_score) + ((1 - preference_slider) * momentum_score)
    
    # Scale position based on the final weighted score
    if final_score > 80:
        return 3.0  # Max size (High Conviction)
    elif final_score > 50:
        return 1.5  # Medium size
    return 1.0      # Base size
