Technical Specifications for Permanence OS Core Agents
1. Architectural Foundation: The Governed Agent Framework
Permanence OS is engineered on the foundational directive of  "No Autonomy Without Audit."  Every agent is classified as a "governed employee," operating within a strictly defined sandbox to eliminate "wandering" and unmapped execution. The system mandates that all mutations are  append-only , ensuring an immutable audit trail and verifiable provenance for every API call and file modification.
Three-Layer Context Architecture
Agent focus and resource consumption are restricted through a hierarchical scoping mechanism:
* Layer 1: CLAUDE.md (The Map):  The root-level authority file. It defines the global floor plan, naming conventions, and the "never touch" list. Every agent session MUST ingest this file first to establish the operational boundaries.
* Layer 2: context.md (Routing Tables):  Directory-specific scoping files located in site/foundation/, agents/, core/, and scripts/. These files provide local routing instructions, explicitly defining which files are relevant to the workspace and which MUST be ignored.
* Layer 3: Workspaces:  Integrated .vscode/ configurations and specialized directories (e.g., docs/design/) that enforce IDE-level file grouping and task-specific workflow templates.
Global Logic Constraints
* Spending Gates:  The system SHALL route all API expenditures through spending_gate.py, enforcing daily budget caps and mandatory timed approvals.
* Canon Compliance:  Logic MUST validate every significant action against the system Canon (base_canon.yaml and brand_identity.yaml).
* Local-First Integrity:  All core governance and logic execution MUST remain on the operator’s machine to eliminate cloud dependencies and maintain the integrity of the Sovereign Gate.
* Test Baseline:  No agent-generated code SHALL be finalized unless the system passes the baseline of  772+ tests across 163 files .
2. Core Agent Specification: The Research Agent
The intelligence gathering workflow is split between high-fidelity ingestion (researcher.py) and intelligence compilation (synthesis_agent.py).
Technical Logic
researcher.py utilizes researcher_adapters.py to interface with external APIs and local document stores. The agent SHALL execute document analysis and web scraping to populate the raw context queue. Once ingestion is complete, synthesis_agent.py is triggered to compile the intelligence into structured entries within the  Synthesis DB .
Trigger Conditions
* Task decomposition sequences initiated by planner.py.
* Direct execution of ingestion scripts (e.g., ingest_sources.py, social_research_ingest.py).
Input/Output Specifications
Input Source,Technical Library/Adapter,Output Format
Web Intelligence,Tavily Search Adapter,Synthesis DB (SQLite)
PDF Documents,pypdf extraction,Markdown Research Briefs
Cloud Documents,Google Drive / Docs API,knowledge_graph/graph.json
Repository Data,GitHub Research Ingest,Chronicle Timeline Entries
Quality Constraints
The  Reviewer Agent  (reviewer.py) SHALL intercept Research Agent outputs to perform "source dominance detection." Any output relying on a singular source or failing to meet evidence coverage thresholds MUST be rejected and rerouted for deeper ingestion.
3. Core Agent Specification: The Coach Agent
The Coach Agent synthesizes behavioral support (therapist_agent.py) and capability development (trainer_agent.py) logic.
Technical Logic
The agent SHALL balance human-centric wellness tracking with system-centric agent training. It monitors the "Identity" of the system (identity.py) to ensure all evolutions align with the operator's mental models and values.
Trigger Conditions
* Health Triggers:  Low health scores generated by health_agent.py.
* Shepherd Cycle:  Weekly reporting intervals managed by the HR Agent (hr_agent.py) based on the "Family Covenant."
Constraints
The Coach Agent is forbidden from proposing architectural changes that violate the  Family Covenant . All proposed system improvements MUST be logged to chronicle/ for historical auditing before implementation.
4. Core Agent Specification: The Finance Agent
The Finance Agent governs the OS's economic pipeline, operating the  "Money First" gate  via money_first_gate.py.
Technical Logic
The agent SHALL manage the revenue pipeline (leads, outreach, and deal events) while simultaneously auditing the cost_tracker.py. Logic MUST evaluate "Cost-Recovery" metrics—balancing real-time API spend against generated revenue—before clearing the Spending Gate for high-cost operations.
Trigger Conditions
* New entries in the revenue_action_queue.py.
* Daily budget cap evaluations and timed approval requests.
* Stripe/Revenue intake synchronization events.
Quality Constraints
* Daily Caps:  Absolute hard-coded spending limits.
* Cost-Recovery Validation:  The agent SHALL block any non-essential API call if the system's cost-recovery ratio falls below defined thresholds.
5. Core Agent Specification: The Content Agent
The Content Agent (social_agent.py) manages multi-platform communication and brand presence.
Technical Logic
The agent operates a multi-platform draft queue for Discord, Telegram, and X. It SHALL utilize promotion_daily.py and the run_comms_loop.sh automation loop to transform Synthesis DB briefs into public-facing content.
Constraints
* Human-in-the-Loop:  Explicitly forbidden from releasing any content to external APIs without an "approved" status in the Sovereign Gate.
* Formatting:  All drafts MUST adhere to the design system (fonts: Sora/Orbitron; colors: Teal/Gold). Emojis are FORBIDDEN; agents SHALL use text labels or SVG references.
6. Core Agent Specification: The Governance Agent
The Governance Agent (Polemarch) represents the supreme oversight layer, utilizing polemarch.py, compliance_gate.py, and reviewer.py.
Technical Logic & Flow
The Governance Agent SHALL perform a three-step risk assessment for all tasks:
1. Intake Sanitization:  reception_agent.py or interface_agent.py cleans inputs and assigns unique ticket IDs.
2. Canon Compliance:  compliance_gate.py evaluates the task against base_canon.yaml and brand_identity.yaml.
3. Risk Routing:  If a task triggers "destructive" or "public-facing" flags, Polemarch SHALL intercept and route it to the Sovereign Gate.
Identity & Security Enforcement
* Design Guardrails:  The agent SHALL enforce the "Never Touch" list, specifically rejecting any UI modifications that use forbidden fonts (Inter, Roboto, Arial, Space Grotesk).
* Security Scanning:  Any proposed code mutation MUST pass a secret_scan.py check (searching for exposed keys) as a pre-push hook.
* Model Routing:  Logic SHALL utilize model_router.py to select the optimal model (Claude, OpenAI, XAI, or Ollama) based on the task's risk profile and cost.
7. Data Integration: The Seven Core Databases
Agents SHALL interact with the storage layers via storage.py and memory.py using the following interaction patterns:
1. Chronicle:   Location:  memory/chronicle/. The historical timeline and published outputs.
2. Episodic:   Location:  memory/episodic/. Append-only JSONL logs capturing every prompt/response for provenance.
3. Working Memory:   Location:  memory/working/. Active task state and temporary execution context.
4. Zero-Point State:   Location:  memory/zero_point_store.json. The persistent master identity and system state.
5. Synthesis DB:   Location:  core/synthesis_db.py (Manager). Compiled intelligence and multi-source research briefs.
6. Knowledge Graph:   Location:  knowledge_graph/graph.json. Entity-relationship mapping across the project.
7. Permanence DB:   Location:  permanence_storage/permanence.db. The master SQLite instance for structured system data and historical persistence.
8. The Sovereign Gate & /REVIEW Process
The Sovereign Gate is the final authority layer where the human operator (Payton Hicks) validates governed actions.
Operational Interface
* Command Center:  Operating at http://127.0.0.1:8000, this dashboard serves as the central operational UI.
* Agent View:  Located at agent_view.html, this provides a  FaceTime-like  interface for monitoring the agent roster and real-time status.
Review Workflow
1. Request:  Actions enter the queue at /api/approvals.
2. Validation:  All actions MUST pass secret_scan.py before appearing in the queue.
3. Visual Status:  Pending actions are visualized as "Waiting for Review" on the dashboard using CSS variables --c1 (Teal) for active status and --rose for high-risk alerts.
4. Final Authority:  No autonomous action SHALL execute without a manual approve command.  Payton Hicks is always the final authority.
